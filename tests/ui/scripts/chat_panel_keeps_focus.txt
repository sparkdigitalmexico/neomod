# chat panel focus retention (fixes a 4.2 regression WH caught): clicking WITHIN the chat
# panel (message area / user list, not just the input box) must KEEP the input focused so
# typing stays in chat; only a click truly OUTSIDE the panel releases focus. before the fix,
# clicking off the input but still inside chat dropped focus and the next keystroke leaked
# into the song browser's type-to-search behind the still-open chat. Chat::updateInput now
# re-focuses the input on any left-press within Chat::isMouseInside() (scoped to the panel, so
# it does NOT re-introduce the global grab that chat_focus_steal guards).
# uses debug_fake_room for a selected channel (#multiplayer) so the message area exists.
# small-chat geometry at 1280x720 (Chat::updateLayout): messages (0,504,768,186), input
# (0,690,768,30), room UI on the right (x>768). assertion-only (the room has avatars/async
# bits, so no ui_trace; the text asserts carry the proof and the empty trace golden is stable).
options_save_on_back 0
debug_fixed_frametime 0.0166667
mouse_raw_input 0
mouse_sensitivity 1
ui_scale_to_dpi 0
windowed_resolution 1280x720
mouse_to 10 10
@wait 30
debug_fake_room
@wait 60
ui_assert visible chat 1
mouse_to 200 700
@wait 3
mouse_down
@wait 3
mouse_up
@wait 3
sendtext hi
@wait 3
ui_assert text chat_input hi
mouse_to 200 600
@wait 3
mouse_down
@wait 3
mouse_up
@wait 3
sendtext more
@wait 3
ui_assert text chat_input himore
mouse_to 1050 250
@wait 3
mouse_down
@wait 3
mouse_up
@wait 3
sendtext zzz
@wait 3
ui_assert text chat_input himore
exit
